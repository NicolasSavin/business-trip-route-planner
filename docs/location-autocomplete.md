# Подсказки городов и станций

## Источник справочника

`LocationRepository` использует компактный локальный справочник основных городов России и станций, которые уже нужны приложению для демонстрационных и провайдерских маршрутов. Архитектура не привязана к этому списку: записи представлены отдельной моделью `LocationRecord`, поэтому официальный справочник Яндекс Расписаний можно загружать заранее и добавлять в репозиторий без изменения публичного API.

API-ключ Яндекс Расписаний не передаётся во frontend. Если провайдер Яндекс Расписаний включён, backend должен использовать только официальный API или заранее скачанный официальный список станций/населённых пунктов. Онлайн-запрос подсказок к неподтверждённым endpoint не используется.

## Endpoint

`GET /api/v1/locations/suggest?q={query}&limit=10`

Ответ содержит массив `items` с `id`, `name`, `display_name`, `type`, `provider_code`, `region` и `country`. Поддерживаемые типы: `city`, `station`, `bus_station`, `railway_station`, `settlement`.

## Алгоритм поиска

Поиск начинается на backend после нормализации запроса. Репозиторий ищет:

1. точное совпадение;
2. совпадение с началом полного названия;
3. совпадение с началом любого слова;
4. совпадение по части названия.

При одинаковой релевантности города и населённые пункты сортируются выше станций. Затем учитывается отображаемое имя, а результат ограничивается параметром `limit`.

## Нормализация

`LocationNormalizer` приводит текст к нижнему регистру, удаляет лишние пробелы, заменяет `ё` на `е` и унифицирует дефисы для поиска. Оригинальное `display_name` сохраняется и возвращается пользователю без искажений.

Поддерживаются распространённые варианты: `СПб` и `Питер` → `Санкт-Петербург`, `Нижний` → `Нижний Новгород`, `Ростов` → `Ростов-на-Дону`.

## Кэш

Для подсказок используется in-memory TTL cache без Redis. Ключ — нормализованный запрос и `limit`, TTL по умолчанию 20 минут, максимальный размер — 256 записей. Ошибки не кэшируются, секреты в кэш не попадают.

## Города и станции

Frontend показывает `display_name`, тип объекта и регион. При выборе сохраняются `id`, `provider_code` и `type`. Ручной ввод остаётся разрешённым: если пользователь не выбрал подсказку, поиск продолжит отправлять текстовые `origin` и `destination` как раньше.

## Ограничения

Локальный справочник компактный и не заменяет полный официальный список Яндекс Расписаний. Для производственного покрытия нужно регулярно обновлять предзагруженный официальный справочник.

## Как обновлять официальный список Яндекс Расписаний

1. Скачать официальный список станций/населённых пунктов из документированного источника Яндекс Расписаний в backend-задаче или maintenance-скрипте.
2. Преобразовать записи к полям `LocationRecord`: стабильный `id`, `name`, `display_name`, `type`, `provider_code`, `region`, `country`, `aliases`.
3. Сохранить подготовленный файл в репозитории или persistent storage backend.
4. Загружать файл один раз при старте приложения, а не на каждый запрос подсказок.
5. Не коммитить и не отдавать frontend API-ключи или приватные параметры доступа.
