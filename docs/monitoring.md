# Monitoring Engine

Monitoring Engine — внутренний слой проверки сохранённых заявок на командировки. Он не отправляет уведомления, не бронирует билеты и не изменяет `SavedSearch`: единственный результат работы — запись истории проверки.

## Архитектура

Пакет backend: `backend/app/monitoring`.

Основные классы:

- `MonitoringEngine` — выполняет проверку одной заявки, получает маршруты, считает снимок состояния и определяет изменения относительно предыдущей записи истории.
- `MonitoringService` — связывает `SavedSearchRepository`, `MonitoringEngine` и `MonitoringHistoryRepository`, предоставляет сценарии `run_one`, `run_all`, `list_history`.
- `MonitoringScheduler` — простой in-process scheduler без Celery, Redis, cron. Умеет запускать одну или все активные заявки, не допускает параллельный запуск одной заявки, ведёт журнал запусков.
- `MonitoringHistory` — модель записи истории одной проверки.
- `MonitoringResult` — модель ответа о запуске проверки.
- `MonitoringPolicy` — настройки проверки и сравнения, включая допуск изменения score.

## Жизненный цикл проверки

1. API или внутренний код вызывает `MonitoringScheduler.run_one(saved_search_id)` или `MonitoringScheduler.run_all()`.
2. Scheduler берёт lock конкретной заявки. Если такая проверка уже идёт, повторный запуск пропускается.
3. `MonitoringService` загружает `SavedSearch` из репозитория.
4. `MonitoringEngine` строит поисковый request из заявки и запускает route search с недоступными маршрутами.
5. Новый снимок сравнивается с последней записью `MonitoringHistory` этой заявки.
6. Создаётся новая запись истории со статусом, длительностью, количеством маршрутов, количеством доступных маршрутов, лучшим score, признаком изменений и summary.
7. `SavedSearch` не меняется.

## Формат истории

Каждая запись `MonitoringHistory` содержит:

- `id` — идентификатор записи;
- `saved_search_id` — идентификатор заявки;
- `checked_at` — время проверки;
- `duration_ms` — длительность проверки;
- `routes_found` — количество найденных маршрутов;
- `available_routes` — количество маршрутов, доступных для группы;
- `best_score` — score лучшего маршрута, если он есть;
- `status` — `success`, `failed` или `skipped`;
- `change_detected` — были ли обнаружены изменения;
- `summary` — человекочитаемое описание результата;
- `changes` — список конкретных изменений;
- `route_ids` — снимок идентификаторов маршрутов для последующего сравнения;
- `free_seats` — агрегированное количество свободных мест по маршрутам.

## Алгоритм определения изменений

Изменением считается любое из событий:

- появились новые route id;
- общее количество маршрутов стало больше;
- количество доступных маршрутов стало больше;
- ранее доступных маршрутов не было, а теперь появились места для группы;
- изменился `best_score` лучшего маршрута больше допуска `MonitoringPolicy.score_tolerance`;
- изменилось агрегированное количество свободных мест.

Если изменений нет, summary равно `Без изменений`.

## API

- `GET /api/v1/monitoring/history/{saved_search_id}` — история проверок заявки.
- `POST /api/v1/monitoring/run/{saved_search_id}` — ручной запуск мониторинга одной заявки.
- `POST /api/v1/monitoring/run-all` — запуск мониторинга всех активных заявок.

## Ограничения этапа

Monitoring Engine на этом этапе только обнаруживает и фиксирует изменения. Уведомления, бронирование, изменение заявки и внешние очереди задач остаются за пределами текущей реализации.
