# Архитектура

## Компоненты

- `backend/app/main.py` создаёт FastAPI-приложение, health-check и подключает маршруты API.
- `backend/app/models/routes.py` содержит Pydantic-модели запроса, сегментов, доступности и ответа.
- `backend/app/providers/base.py` задаёт абстрактный `TransportProvider` для расписаний и транспортных сегментов.
- `backend/app/providers/mock.py` реализует mock-провайдер поездных и автобусных сегментов.
- `backend/app/engine/route_engine.py` координирует построение графа, поиск маршрутов, ранжирование и композиционную проверку доступности.
- `backend/app/availability` содержит независимый Availability Engine: политику проверки, интерфейс `AvailabilityProvider`, mock-провайдер доступности и агрегацию результатов по маршруту.
- `backend/app/services/route_search.py` сериализует доменные маршруты в публичный API-контракт.
- `frontend/app/page.tsx` содержит одну рабочую страницу поиска.

## Маршрутный движок

Движок получает сегменты через интерфейс провайдера, а не напрямую из mock-данных. Алгоритм:

1. Получить сегменты на дату поездки с учётом разрешённых видов транспорта.
2. Построить транспортный граф.
3. Найти прямые маршруты и маршруты с разрешённым количеством пересадок.
4. Проверить, что время между прибытием первого сегмента и отправлением второго не меньше `minimum_transfer_minutes`.
5. Рассчитать пересадки, оценку, объяснения, предупреждения и преимущества.
6. Передать найденные `RouteOption` в `AvailabilityEngine`, который проверяет наличие мест по сегментам и прикрепляет необязательный блок `availability`.
7. Сохранить обратную совместимость API: публичный поиск возвращает маршруты, пригодные для группы, и не требует от frontend использовать новый блок.

Route Engine не реализует финальные правила доступности мест. Он может использовать данные сегмента только как часть графа и ранжирования; окончательное решение о пригодности маршрута для группы принимает Availability Engine.

## Availability Engine

Availability Engine отвечает за вопрос «можно ли отправить N сотрудников по уже найденному маршруту». Он:

- принимает `RouteOption` и `AvailabilityPolicy`;
- проверяет каждый `TransportSegment` через отдельный `AvailabilityProvider`;
- собирает `SegmentAvailability` и `RouteAvailability`;
- считает минимальное число доступных мест среди участков;
- добавляет причины недоступности, предупреждения и признаки устаревших данных;
- не изменяет исходный `RouteOption`.

Подробности описаны в [`docs/availability-engine.md`](availability-engine.md).

## Будущие провайдеры

Реальные интеграции расписаний должны реализовывать `TransportProvider`. Реальные интеграции инвентаря мест, например будущий `RzdAvailabilityProvider`, должны реализовывать отдельный `AvailabilityProvider`. Это позволяет добавить провайдеры поездов, автобусов, агрегаторов или инвентаря мест без переписывания поиска маршрутов и API-сервиса.
