# Архитектура

## Компоненты

- `backend/app/main.py` создаёт FastAPI-приложение, health-check и подключает маршруты API.
- `backend/app/models/routes.py` содержит Pydantic-модели запроса, сегментов и ответа.
- `backend/app/providers/base.py` задаёт абстрактный `TransportProvider`.
- `backend/app/providers/mock.py` реализует mock-провайдер поездных и автобусных сегментов.
- `backend/app/services/route_search.py` содержит маршрутный движок.
- `frontend/app/page.tsx` содержит одну рабочую страницу поиска.

## Маршрутный движок

Движок получает сегменты через интерфейс провайдера, а не напрямую из mock-данных. Алгоритм:

1. Получить сегменты на дату поездки с учётом разрешённых видов транспорта.
2. Найти прямые сегменты `origin → destination`.
3. Если разрешена одна пересадка, перебрать пары сегментов `origin → transfer` и `transfer → destination`.
4. Проверить, что время между прибытием первого сегмента и отправлением второго не меньше `minimum_transfer_minutes`.
5. Рассчитать общее время, город пересадки, длительность пересадки и количество пересадок.
6. Пометить маршрут доступным для группы только если на каждом сегменте `available_seats >= passengers`.
7. Отсортировать маршруты по количеству пересадок и общему времени.

## Будущие провайдеры

Реальные интеграции должны реализовывать `TransportProvider`. Это позволит добавить провайдеры поездов, автобусов или агрегаторов без переписывания `RouteSearchService`.
